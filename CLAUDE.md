# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Self-hosted, **single-user** health dashboard. It joins **Garmin Connect** (pulled hourly via `python-garminconnect`) and **Apple Health** (pushed via the Health Auto Export iPhone app) into a daily 0‚Äì100 condition score, evidence-based wellbeing alerts, and Anthropic Claude coaching. Served privately over a Tailscale tailnet ‚Äî no public ingress. It is explicitly **not a medical device**; alerts are framed conservatively. See `README.md` and `SPEC.md` for product framing; `docs/superpowers/specs/*.md` for per-feature design rationale.

```bash
# Backend (Python 3.12, uv)
cd backend
uv venv --python 3.12 .venv
uv pip install --python .venv/bin/python -e '.' --group dev
.venv/bin/python -m pytest                      # all tests
.venv/bin/python -m pytest tests/test_forecast.py::test_name -q   # single test
.venv/bin/python -m ruff check app/ tests/      # lint (line-length 100, py312)

# Frontend (React + Vite)
cd frontend
npm install
npm run dev      # http://localhost:5173, proxies /api and /ingest to :8000
npm run build    # tsc -b && vite build
npm run lint     # eslint

# Whole stack (Docker: backend + frontend/nginx + tailscale sidecar)
docker compose up -d --build
bin/up.sh        # same, but resolves secrets from 1Password into .env.runtime

# CLI (inside the backend container)
docker compose exec backend python -m app.cli <garmin-login|sync-garmin|recompute [YYYY-MM-DD]|regenerate-advice [YYYY-MM-DD]>
```

`ruff` ignores `RUF001/002/003` (they false-flag full-width Japanese punctuation) and `B008` (FastAPI `Depends()`). `mypy` is non-strict.

### Backend layering (`backend/app/`)

Keep these layers distinct ‚Äî API handlers stay thin, business logic lives in `scoring/`:

- **`ingest/`** ‚Äî external data in. `garmin_client.py`/`garmin_sync.py` (Garmin pull, MFA login persisted), `hae_parser.py`/`hae_writer.py` (Apple Health JSON ‚Üí DB), `pressure_history.py` (barometric data for migraine triggers).
- **`models/health.py`** ‚Äî the *entire* SQLAlchemy schema in one file (`MetricSample`, `SleepSession`, `HrvDaily`, `BodyBatteryDaily`, `DailyScore`, `UserProfile`, `LlmComment`, plus food/caffeine/alcohol/migraine/learning/notification/fitness-test tables). `db.create_all()` runs on startup ‚Äî **there are no migrations**; schema changes must stay additive/compatible with existing SQLite files.
- **`scoring/`** ‚Äî all the math, pure-ish and unit-tested. This is the heart of the app. `recompute.py:recompute_for_date()` is the orchestrator that reads a day's raw rows, computes each subscore, and writes one `DailyScore`. Subscores combine via `composite.py` using configurable weights. `achievement.py` defines the **achievement-curve** model: each metric maps to 0‚Äì100 by distance from an ideal band (`band_achievement`) or a one-sided floor/goal (`upper_achievement`), not by raw thresholds.
- **`scoring/wellbeing_alerts.py`** ‚Äî rule-based medical alerts grounded in published research (sleep debt, HRV decline, SpO2 desaturation, MOH risk, etc.).
- **`llm/`** ‚Äî `client.py` gathers a rich context bundle (recent workouts, caffeine, migraine, focus, pressure, prior advice) and calls Claude (`prompts.py`); `quiz.py` and `food.py` are feature-specific LLM calls. Model is `settings.llm_model` (default `claude-sonnet-4-6`); regenerations capped per day.
- **`notifications/`** ‚Äî Web Push. `engine.py` decides what's due, `push.py` sends via VAPID (keys generated by `cli_vapid.py`), `service.py:notification_tick_job` is the scheduler entry.
- **`api/`** ‚Äî one FastAPI router module per feature, all registered in `main.py:create_app()`. `dashboard.py` is the primary aggregate read.
- **`scheduler.py`** ‚Äî APScheduler cron jobs (all cron exprs are `settings`-configurable): `garmin_sync`, `recompute_today`, `morning_advice`, `baseline_refresh` (currently a no-op; baselines computed on demand), `notification_tick`. Jobs lazy-import heavy deps so unit tests stay light.
- **`config.py`** ‚Äî `Settings(BaseSettings)`, the single source of all tuning. Two distinct kinds of constants, kept separate by design: **clinical/physiological** (fixed for everyone ‚Äî SpO2 ‚â•95%, ACWR 0.8‚Äì1.3) vs **personal targets** (per-user env overrides ‚Äî `TARGET_WEIGHT_KG`, `USER_AGE`, Karvonen HR inputs, caffeine pharmacokinetics, bath/meal timing). The committed values are an *example profile*; real values come from env. `app_tz` defaults to `Asia/Tokyo` and date logic is TZ-aware (`scoring/timewindow.py:app_today()`).

### Frontend (`frontend/src/`)

React + Vite + TanStack Query + Tailwind + Recharts. Routing is **hash-based** (`App.tsx`): `#debug` ‚Üí `DebugPage`, else `TodayPage`. No router library. `lib/api.ts` holds all typed fetch wrappers and shared types; `components/` is feature-panel-per-file mapping closely to backend `api/` endpoints. `public/push-sw.js` is the Web Push service worker. Many UI strings and code comments are in Japanese ‚Äî match that.

## Conventions

- **Respond in Japanese** (code identifiers stay English; comments are frequently Japanese).
- Correctness details that look like bugs but aren't: **sleep midpoint** uses *circular* statistics (`scoring/circadian.py`) because it's a cyclic quantity that crosses midnight; **medication-overuse risk** counts distinct medication-days (ICHD-3), not doses. Don't "simplify" these to linear math.
- When adding a tuning constant, decide clinical-vs-personal and put it in `config.py` accordingly ‚Äî don't hardcode in scoring modules.
- New scoring logic gets a unit test in `backend/tests/` (the existing suite tests `scoring/` heavily and is the fast feedback loop ‚Äî no DB or network needed for most).
- Secrets resolve via 1Password (`bin/up.sh` ‚Üí `.env.runtime`) or a plain `.env`. See `memory/deploy-mechanism.md` for the production deploy path.
